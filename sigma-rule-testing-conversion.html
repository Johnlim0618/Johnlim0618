Sigma Rule Testing and Conversion

This project focused on understanding how detection logic is written in Sigma and how that logic translates into SIEM-specific searches and alerting. It helped me build stronger detection thinking and a better understanding of portable rule content.

Overview

Sigma provides a standardized way to write detection logic across platforms. I used this project to practice reading Sigma rules, mapping them to ATT&CK techniques, and thinking through how those rules would need to be translated into SIEM-friendly searches.

Objectives

  • Learn how Sigma rules are structured
  • Understand how portable detection logic becomes platform-specific alerting
  • Improve my ability to reason about false positives and tuning

Tools Used

  • Sigma rules
  • Splunk / SPL concepts
  • MITRE ATT&CK
  • Windows logs

Environment

I reviewed Sigma rules in the context of Windows-based detections and used lab telemetry as a reference point to think through how those detections would behave in a SIEM environment.

What I Did

  • Reviewed the structure of Sigma detections including logsource, selection, and condition
  • Mapped rule logic to Windows activity and ATT&CK techniques
  • Thought through how Sigma content would convert into SIEM-specific queries
  • Evaluated what kinds of activity might cause false positives

Key Findings

  • Portable rules are useful, but local environment context still matters
  • Detection logic needs tuning to avoid noisy or overly broad alerting
  • Understanding the underlying telemetry is just as important as understanding the rule itself

Example Artifact

title: Suspicious PowerShell Encoded Command
id: 7f0f4e01-1111-2222-3333-444455556666
status: experimental
logsource:
  product: windows
  service: powershell
detection:
  selection:
    ScriptBlockText|contains:
      - "EncodedCommand"
      - "FromBase64String"
  condition: selection
level: high

What I Learned

This project helped me understand that detection writing is not just about spotting “bad strings.” It is about understanding telemetry, platform differences, and how rules behave once they are actually deployed.
